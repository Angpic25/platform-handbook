# Issue Management

The Issues tab within each touchpoint provides a complete issue management system. The client can search, filter, sort, and export accessibility issues, and request expert retests.

<video aria-hidden="true" tabIndex={-1} src="/images/video-issue-management.mp4" autoPlay loop muted playsInline controls style={{ width: '100%', borderRadius: '8px' }} />

## How it works


- **Search bar** to find issues by numeric ID or text description (minimum 3 characters).
- **Advanced filters** for severity, status, comments, mitigation, type, WCAG criteria, and pages. Quick filter presets are available for Open issues, Critical issues, and Recurring issues.
- **Sortable columns** including ID, WCAG criteria, Severity, Status, Comments, Mitigation, and Type.
- **Export** the full issue list (or filtered subset) as CSV or Excel.
- **Ask for a retest** button to request an expert review of all fixes applied to the touchpoint.

## Issue status lifecycle

The platform uses four statuses to track the progress of each issue:

<table>
  <caption>Issue status lifecycle: statuses, who sets them, and what they mean</caption>
  <thead>
    <tr>
      <th scope="col">Status</th>
      <th scope="col">Who sets it</th>
      <th scope="col">What it means</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>Open</strong></td>
      <td>System (default)</td>
      <td>The issue has been identified by an expert but not yet addressed</td>
    </tr>
    <tr>
      <td><strong>Fixed</strong></td>
      <td>The client</td>
      <td>The client's development team has implemented a fix and marked the issue accordingly</td>
    </tr>
    <tr>
      <td><strong>Resolved</strong></td>
      <td>AccessiWay expert</td>
      <td>After a retest, the expert has confirmed that the fix meets accessibility requirements</td>
    </tr>
    <tr>
      <td><strong>Dismissed</strong></td>
      <td>The client</td>
      <td>The client has decided to archive or ignore the issue</td>
    </tr>
  </tbody>
</table>

The key distinction is between Fixed and Resolved. Only the client can mark an issue as Fixed (indicating their team has implemented a change). However, only an AccessiWay expert can mark an issue as Resolved, which happens after a retest confirms the fix is effective. This ensures that compliance improvements are always validated by a qualified expert.

### Lifecycle flow

The typical lifecycle follows this path:

**Open → Fixed → Resolved**

The client's team fixes the issue and marks it as Fixed. Then, when the client requests a retest, an AccessiWay expert reviews the touchpoint and confirms the fix by setting the status to Resolved.

Alternatively, the client can dismiss an issue at any point:

**Open → Dismissed**

### Identify and filter issues

The client opens the Issues tab and uses filters to focus on the most critical or relevant issues.

### Fix and mark as Fixed

The development team implements the fix. The client then marks the issue as Fixed in the platform.

### Request a retest

The client clicks "Ask for a retest" to request expert validation. The AccessiWay expert reviews all fixed issues and sets confirmed ones to Resolved.
