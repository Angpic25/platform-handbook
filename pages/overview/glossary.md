# Glossary

| Term | Definition |
|------|------------|
| **Touchpoint** | A website or digital application subject to accessibility analysis |
| **Compliance level** | Percentage of alignment with WCAG standards — an improvement indicator, not a legal certification |
| **Open issue** | An accessibility issue identified but not yet addressed |
| **Fixed issue** | An issue corrected by the dev team but not yet validated by an expert |
| **Resolved issue** | An issue corrected and expert-validated through a retest |
| **Dismissed issue** | An issue archived or ignored — no action required |
| **Recurring issue** | An issue that appears on multiple pages of a touchpoint |
| **Unique issue** | An issue specific to a single page |
| **Mitigation** | A temporary fix applied by the AccessiWay widget while the team works on a permanent code-level fix |
| **Retest** | An expert review confirming that fixes have been correctly implemented (max 1 per touchpoint) |
| **WCAG** | Web Content Accessibility Guidelines — the international accessibility standard (version 2.2 in the platform) |
| **EN 301 549** | European standard for accessibility requirements for ICT products and services |
| **User flow** | A typical user navigation path within the website (e.g. "About Us", "Internet Banking") |
| **Critical severity** | Issues that block essential user tasks — must be fixed urgently |
| **High severity** | Issues that affect main navigation — should be prioritized |
| **Low severity** | Minor usability issues — can be addressed later |
| **2FA** | Two-Factor Authentication for secure platform login |
| **User impact** | A prioritization lens that ranks issues by how much they affect real users, rather than by raw volume |
